# Chavruta.AI — Technical Decisions (ADR)

החלטות טכניות נעולות, עם נימוק וחלופות. כל החלטה משרתת שני יעדים: **התחרות (Nebius)**
וגם **הרצה offline מקומית**.

---

## D1 — מודל Embedding: `BAAI/bge-m3`
**ההחלטה:** להחליף את `bge-small-en-v1.5` ב-**`BAAI/bge-m3`**.

**למה:**
- **רב-לשוני אמיתי** — תומך עברית *ו*אנגלית (פותר את דרישה #5 ואת באג ה"אנגלית בלבד").
- **Hybrid** — dense + sparse + multi-vector באותו מודל → אחזור מדויק יותר.
- קונטקסט ארוך (8192 טוקנים) — טוב למקורות ארוכים (רמב"ן/אור החיים).
- **רץ מקומית** על CPU/GPU → תומך בפרופיל ה-offline (#8).

**חלופות שנשקלו:** `intfloat/multilingual-e5-large` (טוב, פחות חזק מ-bge-m3 ב-hybrid);
`dicta-il` (עברית-ייעודי, אבל חלש באנגלית — נפסל בגלל הדו-לשוניות).

---

## D2 — Vector Store: `Qdrant`
**מה Sefaria עשו:** **Neo4j** — שני מופעים: אחד ל-embeddings (חיפוש וקטורי), אחד
ל**גרף ידע** (קשרים בין מקורות, מטיילים בו + דירוג PageRank). Neo4j אִפשר לשלב וקטור+גרף
באקוסיסטם אחד.

**ההחלטה שלנו:** **Qdrant**, לא Neo4j.

**למה:**
- **רץ גם מקומי (embedded) וגם כשרת** — אותו קוד ל-offline ול-Nebius. Neo4j כבד מדי ללפטופ.
- מהיר, ו-**סינון מטא-דאטה מצוין** (סנן לפי ספר / מפרש / סוג מקור) — קריטי לדירוג שלנו.
- את ה"מקורות הקשורים" (מה שהגרף נתן ל-Sefaria) נשיג מ**נתוני הקישורים של ספריא**
  (מחושבים מראש) — בלי DB גרף מלא. אפשר להוסיף גרף בהמשך אם צריך.

**סדר (מתוקן):** עוברים ל-Qdrant **מיד**, לא דרך ChromaDB. התורה (15,274 צ'אנקים) *מתחת*
למגבלת ה-~20K של Qdrant embedded → ה-MVP רץ ב-Qdrant מקומי/offline. **אותו** אוסף Qdrant
גדל ל-server mode (Docker) כשמוסיפים את שאר ספריא — אפס מיגרציה. ChromaDB הקיים נשאר רק
כ-fallback/השוואה (`data/chroma_db_old_bge_small`).

**זרימת אינדוקס:** `embed_corpus_gpu.py` (GPU, פלט וקטורים גנרי) → `load_to_qdrant.py`
(טעינה מקומית, בלי הטמעה חוזרת). זמן-ריצה: הטמעת השאלה בלבד על CPU (~3 שנ').

## D3 — מודל הגנרציה (Generation)
**תפקיד המודל:** הוא ה"קול", לא ה"זיכרון". הוא מקבל את המקורות שאוחזרו + השאלה,
ו**מנסח** תשובה. הוא **לא צריך לדעת תורה** — RAG מספק את העובדות. הוא צריך:
1. להבין מקור עברי, 2. לנמק/לסנתז, 3. לכתוב שוטף בעברית+אנגלית, 4. לציית להוראת העיגון
("השתמש רק במקורות שסופקו").

**ההחלטה — לפי פרופיל:**
- ☁️ **Nebius (תחרות):** מודל instruct **גדול ורב-לשוני** דרך serverless inference —
  למשל Qwen גדול (72B-class) או Llama-70B. הסקה חזקה + עברית סבירה + per-token.
- 💻 **Local (offline):** **`granite4.1:3b`** (IBM Granite, Apache-2.0) דרך Ollama.

**ממצאים אמפיריים (qwen3:4b מול granite4.1:3b על אותו פרומפט מעוגן):**
- **qwen3:4b** — מבין מדויק, אבל מודל-thinking: שופך חשיבה ל-content ונקטע לפני התשובה. ❌
- **granite4.1:3b** — **לא-thinking → תשובה נקייה וישירה** ✅, RAG-native, קטן (~2GB). אבל עברית
  מחוספסת ודיוק נמוך יותר ב-3B. **נבחר כברירת מחדל מקומית** כי הגשה נקייה > הבנה כלואה.
- 🐛 **באג שתוקן:** ה-`SYSTEM_PROMPT` הישן ציווה "ענה תמיד בעברית" → תוקן לדו-לשוני
  ("ענה בשפת השאלה"). זה שיפר את language-following לכל מודל.

**תובנה:** מודלים קטנים (3-4B) שוליים לאיכות תורנית גבוהה → הגנרציה ה**איכותית** שייכת ל-Nebius.
Granite הוא fallback מקומי/offline טוב. חלופה לעברית: DictaLM 2.0.

## D4 — דו-לשוניות (עברית + אנגלית) — דרישה #5
המערכת **חייבת** לתמוך בשתי השפות מקצה לקצה: אחזור (bge-m3), גנרציה (מודל רב-לשוני +
prompt "ענה בשפת השאלה"), ותצוגת מקור עברי+תרגום אנגלי. זו דרישת-על על כל ההחלטות.

---

## D5 — ליבה אגנוסטית לפריסה (Offline-capable) — דרישה #8
**ההחלטה:** כל רכיב חיצוני (LLM, embedder, retriever, store) מאחורי **ממשק אחיד**,
ונבחר דרך **config**. כך אותו קוד רץ ב-☁️ Nebius וב-💻 לפטופ בלי אינטרנט.

**מה דורש רשת ומה התחליף ל-offline:**
| online | תחליף offline |
|--------|----------------|
| Sefaria Linker API | קישורים שהורדו מראש מ-Sefaria export |
| Nebius inference | Qwen מקומי דרך Ollama |
| הורדת הקורפוס | חד-פעמי — אחרי זה הכל מקומי |

---

## D6 — אחזור היברידי (Hybrid Retrieval)
שילוב שלוש שיטות (כמו Sefaria): (1) וקטורי (bge-m3), (2) Sefaria Linker API לזיהוי
הפניות מפורשות, (3) הרחבה דרך קישורי ספריא. דירוג: דמיון + סיווג מקור ראשי/משני.
